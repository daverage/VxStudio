Here is the product spec.

This keeps the current realtime system intact, adds a slow global loudness model, and introduces a true offline mode without turning the whole product inside out.

# VX Leveler V3 Specification

## Realtime Plus Global Memory Plus Offline Analysis

## 1. Product Goal

VX Leveler must support three operating behaviours:

### A. Vocal Rider

Realtime, phrase-aware, editorially willing to protect speech or vocal clarity.

### B. Mix Leveler Realtime

Realtime, transparent, stabilizes section loudness and controls spikes without sounding compressed.

### C. Mix Leveler Offline

File-aware, analyzes the track ahead of playback or render, builds a loudness map, and applies more coherent end-to-end levelling.

This is one product with two timing models:

* realtime intelligence
* offline intelligence

The core DSP remains shared where sensible, but target generation must differ.

---

# 2. Design Principles

## Keep

* current momentary / short / baseline style realtime system
* fast spike clamp
* slow ride behaviour
* bounded giveback
* vocal and mix mode separation

## Add

* slow global loudness memory in Mix Leveler realtime mode
* optional offline scan mode for file-aware targeting

## Do not do

* do not replace current realtime logic with a purely offline architecture
* do not force offline analysis for live or normal playback
* do not let offline assumptions leak into Vocal Rider mode

---

# 3. New Operating Modes

## Mode 1: Vocal Rider

No global track loudness model.
No offline analysis required.
Uses existing vocal logic, phrase context, and lookahead.   

## Mode 2: Mix Leveler Realtime

Uses:

* current rolling measurements
* new slow global loudness model
* blended target behaviour that becomes smarter over time

## Mode 3: Mix Leveler Offline

Uses:

* pre-analysis pass
* section loudness map
* precomputed global target curve
* same downstream gain engine, but target source changes

---

# 4. Current Realtime System to Preserve

The existing Mix Leveler already has the right local structure:

* fast loudness-like view
* slower section view
* slower recent baseline
* spike clamp
* giveback / normalization recovery

That local system remains the foundation. The new work is about adding a slower memory layer above it, not replacing it. The DSP already separates slow gain behaviour from faster taming behaviour, which is the right starting point. 

---

# 5. New Realtime Global Loudness Model

## Purpose

Give Mix Leveler a growing understanding of the track as a whole, without requiring a full pre-scan.

## Core idea

The plugin must maintain a very slow estimate of:

* what this track’s normal loudness seems to be
* what range of section loudness is typical
* how confident it is in that estimate

## New internal measurements

Add the following in Mix Leveler DSP state:

```cpp
float globalShortEstimateLU = 0.0f;
float globalMedianEstimateLU = 0.0f;
float globalUpperEstimateLU = 0.0f;
float globalConfidence = 0.0f;
float globalDynamicRangeEstimateLU = 0.0f;
```

These are conceptual names. Final naming can follow project style.

## Inputs

Update these using the existing short-term loudness style measurement generated by the realtime path.

## Time behaviour

These estimates must evolve slowly:

* minimum useful memory horizon: 10 seconds
* preferred horizon: 20 to 60 seconds
* confidence should rise from 0 toward 1 as enough material is observed

## Behaviour

At the start of playback:

* low confidence
* plugin behaves mostly like current local realtime version

As playback continues:

* confidence rises
* target gradually shifts from local-only baseline toward local-plus-global baseline

---

# 6. Realtime Blend Logic

## New target structure

Current local logic is roughly:

* momentary = immediate loudness
* short = passage loudness
* baseline = recent normal

Add:

* global baseline = long memory normal

## New target rule

For Mix Leveler realtime:

```text
target_local = recent baseline
target_global = long-memory baseline
blend = globalConfidence
target_final = lerp(target_local, target_global, blendAmount)
```

Where `blendAmount` is shaped, not linear.

Recommended:

* confidence below 0.2: little or no global influence
* confidence 0.2 to 0.7: gradual influence
* confidence above 0.7: stable global weighting

## Why

This avoids the early-track problem where the plugin acts too certain before it has seen enough material.

---

# 7. Realtime Gain Rules

## Keep three-rule structure

### Rule 1: Slow Ride

Compare current short-term loudness against `target_final`.

If section is above target:

* apply slow downward gain

If section is below target:

* allow upward recovery, but slower and smaller than downward control

### Rule 2: Fast Spike Clamp

Compare momentary against short-term.

If momentary overshoots short-term by threshold:

* apply fast downward attenuation

This remains separate from the slow ride.

### Rule 3: Bounded Giveback

After ride and spike control:

* estimate whether processed output has become too low overall
* return a limited amount of level if safe
* reduce or suppress giveback when spike clamp is active

This keeps the system from sounding unnecessarily deflated.

---

# 8. Global Model Rules

## 8.1 Confidence Model

Add a confidence meter based on observed programme duration and diversity.

Confidence rises with:

* elapsed audio time
* number of valid short-term windows observed
* stability of global estimate

Confidence must rise slowly and never jump.

Suggested design:

* 0.0 at start
* near 0.5 after 15 to 20 seconds
* near 1.0 only after substantial exposure

## 8.2 Estimate Robustness

Do not use simple averaging only.

The global model should prefer robust measures:

* rolling percentile
* clipped mean
* median-like estimate
* low/high bounds for dynamic range

Reason:
simple average is too sensitive to isolated loud passages.

## 8.3 Start-of-Track Policy

Until confidence is high enough:

* favour local baseline
* severely limit upward normalization
* do not assume the track is globally quiet just because the intro is quiet

This directly addresses the quiet-intro loud-ending case.

---

# 9. Offline Mode

## Purpose

Allow true file-aware levelling when the host or workflow supports it.

## User promise

Offline mode should:

* analyze the entire file or region before processing
* build a track-level loudness model
* produce more coherent start-to-end levelling than the realtime path

## Scope

Offline mode applies to Mix Leveler only.

Do not make Vocal Rider dependent on offline analysis.

---

# 10. Offline Analysis Pass

## Output of analysis pass

The offline scanner must produce:

### Global stats

* integrated loudness estimate
* short-term loudness distribution
* estimated dynamic range
* loudest section estimate
* quietest sustained section estimate

### Section map

For time slices across the file, store:

* momentary loudness proxy
* short-term loudness proxy
* local baseline
* section confidence
* optional transient density / spike density

### Target curve

Generate a target loudness curve for the whole file, not just one global number.

This curve should be:

* smooth
* section-aware
* conservative
* not a flattened straight line

---

# 11. Offline Processing Model

## Rule

Offline mode still uses the existing ride and spike engine, but the target is no longer inferred live.

Instead:

```text
current_time -> consult precomputed target map
slow ride -> compare current short-term loudness to target map
spike clamp -> still local and fast
giveback -> still bounded by headroom and spike activity
```

This is important because it lets you reuse most of the DSP character while improving the intelligence of the target.

---

# 12. Offline Target Generation Rules

## 12.1 Preserve Structure

Offline mode must not flatten intended arrangement dynamics.

Examples:

* quiet intro can stay quieter
* verse and chorus can differ
* breakdown can remain lower than climax

So the target curve must not force everything to one level.

## 12.2 Correct Excess, Not Intent

Offline mode should reduce:

* accidental section jumps
* inconsistent recording level
* rogue spikes
* over-loud passages relative to the whole

Offline mode should preserve:

* musical contour
* section contrast
* phrase shape
* transient life

## 12.3 Target Curve Constraints

The precomputed target must obey:

* max downward movement rate
* max upward movement rate
* section deadband
* dynamic range floor so the result never becomes flat

---

# 13. New Internal Components

## A. Global Loudness Tracker

New class, Mix Leveler only.

Suggested responsibility:

* consume short-term loudness observations
* maintain long-memory estimates
* expose target and confidence

Suggested interface:

```cpp
class GlobalLoudnessTracker {
public:
    void prepare(double sampleRate, int maxBlockSize);
    void reset();
    void update(float shortLoudnessLU, float momentaryLoudnessLU, bool validWindow);
    float getGlobalBaselineLU() const noexcept;
    float getGlobalUpperLU() const noexcept;
    float getConfidence() const noexcept;
};
```

## B. Offline Analyzer

New analysis-only component.

Suggested responsibility:

* scan audio source or rendered buffer
* generate loudness map and target curve

Suggested outputs:

* vector of section descriptors
* vector of target values
* summary stats

## C. Target Provider Abstraction

DSP should stop caring whether target came from:

* local realtime model
* realtime global blend
* offline precomputed map

Create a small abstraction that supplies target values.

---

# 14. DSP Refactor Requirements

## 14.1 Separate Target Generation from Gain Application

Right now the DSP appears to mix decision generation and gain behaviour tightly. The next version should split:

* measurement
* target generation
* gain computation
* output normalization

The processor already separates detector and DSP at a high level, but Mix Leveler target generation needs cleaner internal boundaries.  

## 14.2 Add Mode-Specific Path

General mode must have a dedicated target path that can accept:

* local-only target
* local-plus-global target
* offline map target

Vocal mode must keep its own existing content-aware path.

# 15. UI / UX Spec

## Existing visible modes

Keep:

* Vocal Rider
* Mix Leveler

Do not expose too much internal complexity.

## Add one new user-facing option

For Mix Leveler only:

### Analysis Mode

Options:

* Realtime
* Smart Realtime
* Offline

Alternative simpler wording:

* Live
* Adaptive
* Analyze Track

Best product wording depends on host context, but the distinction must be clear:

* Realtime = no file scan
* Smart Realtime = learns over time
* Offline = analyze first

## Status messaging

Display status such as:

* Learning track
* Track understood
* Analysis ready
* Offline map active

This helps users understand why the start of the track may behave differently from later sections.

---

# 16. Host / Workflow Behaviour

## Realtime playback

Default to Smart Realtime in Mix Leveler:

* no pre-scan
* global model learns over time

## Offline render or explicit analyze action

If host supports offline preparation or the user triggers analysis:

* run offline scan
* cache target curve
* process with offline target

## If offline analysis is unavailable

Fallback cleanly to Smart Realtime.

---

# 17. Parameter Behaviour

## Level

Defines how strongly the track is pulled toward the target.

In Mix Leveler:

* affects slow ride depth
* affects upward recovery limit
* affects offline target adherence strength

## Control

Defines how aggressively spikes and overs are contained.

In Mix Leveler:

* affects spike clamp threshold / strength
* affects giveback suppression when peak control is active

## Hidden parameters

Do not expose these initially:

* global confidence rise rate
* global memory window
* upward recovery cap
* offline curve smoothness
* section deadband width

These should be tuned internally first.

---

# 18. Safety Rules

## Must always hold

* no gain explosion
* no unstable target jumps
* no abrupt image shifts
* no unbounded upward recovery
* no offline curve discontinuities
* no new allocations in audio thread

## Offline-specific

* analysis must be deterministic
* cached target map must match file length and sample rate assumptions
* invalid cache must be discarded safely

---

# 19. Validation Criteria

## Smart Realtime success criteria

On a track with quiet intro and louder ending:

* early section is not over-lifted
* later sections are stabilized more intelligently than current version
* plugin sounds more informed over time, not more active

## Offline success criteria

On the same track:

* the whole track feels more coherent end to end
* quiet intro can remain quieter if musically justified
* loud ending is controlled without obvious compression
* level contour sounds intentional, not flattened

## Regression criteria

Current good behaviour must remain:

* fast spikes still get controlled
* plugin still feels dynamic
* no obvious pumping
* no “always moving” feel

---

# 20. Recommended Implementation Order

## Phase 1

Refactor Mix Leveler internals so target generation is a separate subsystem.

## Phase 2

Add Smart Realtime global loudness tracker and confidence-based blending.

## Phase 3

Tune Smart Realtime until it clearly beats the current build on long-form material.

## Phase 4

Add offline analyzer and target map generation.

## Phase 5

Connect offline target map to the same downstream ride / spike engine.

## Phase 6

Add user-facing analysis mode and state messaging.

---

# 21. Final Product Definition

After this change, VX Leveler becomes:

### Vocal Rider

A realtime, content-aware vocal protection and riding tool.

### Mix Leveler

A transparent loudness stabilizer with two intelligence levels:

* Smart Realtime: learns the track over time
* Offline: understands the full track before processing

That is a clean, defensible product story.
