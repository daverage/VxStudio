# Lessons

- For shared VX parameter displays, never present a neutral-centered control as a raw `0..1` or as if `50%` were unity by default; if the product meaning is unity at centre, the text and mapping must read that way explicitly, such as `50% .. 150%` with `100%` at the midpoint.
- When a user asks whether a plugin concept can be built, do not anchor too early on the illustrative example signal pair; restate and design around the broader outcome contract first, then narrow the DSP and wording only after confirming the real capture scenario.
- When a user clarifies that the goal is overall perceived level consistency rather than recovery of one source within a mix, re-center the plugin contract around whole-track levelling immediately; source-preservation heuristics should become guard rails, not the headline behavior.
- For shared history/trace widgets, never fabricate “valid” future samples by repeating the last value across unused history; carry explicit coverage/count metadata into the view so the drawing only spans real collected history.
- When an adaptive leveller still feels like broad gain reduction after multiple tuning passes, stop tweaking continuous weights and add an explicit state/decision layer; detector metrics alone do not create intentional behaviour unless the runtime commits to distinct modes.
- When a levelling product needs to serve both speech-masked material and generic program audio, do not “turn down” the speech-aware heuristics into a pseudo-general mode; split the decision logic into separate `Voice` and `General` engines that share DSP primitives but follow different behavioural contracts.
- For adaptive levelers, “more aggressive at the top of the dial” must mostly buy stronger selective downward control and faster intervention, not simply more upward range or more total gain motion; otherwise the plugin looks busier but the real-file level spread can get worse.
- When a smooth adaptive rider plateaus short of the product goal, add a separate intervention lane that activates only in the real override region and only on the offending moments; trying to fake that with one gain law usually makes the plugin more active without making it smarter.
- When pushing an adaptive leveler toward a more radical override law, treat the real-file measurement harness as the gatekeeper: stronger dB clamps and peak-biased triggers can look smarter on paper but still make the target capture worse, so keep only the most aggressive branch that proves itself on the user file.
- When a larger architecture rewrite is conceptually better but measurably worse on the target capture, keep the clearer product wording if it helps, but revert the DSP itself; do not strand the user on a weaker “future direction” build just because the architecture story is nicer.
- For mixed-track vocal levelling, do not assume a speech-band anchor automatically helps; if the loudness metric still judges the result worse, revert it and look for smarter decision logic rather than adding more internal reference envelopes.
- For suite-wide `Voice`-mode work, build a small real-world speech corpus early instead of tuning only against one user file and synthetic tests; architecture decisions that look plausible on one example can fail immediately once exposed to multiple real recordings.
- The first shared `Voice`-mode improvements should usually enter through the framework analysis layer, not through a brand-new per-product DSP branch; enriching detector evidence with reusable vocal context can produce small but real gains across multiple files without destabilising the proven processing core.
- Once a shared vocal-context layer exists, roll it out first through products that already consume structured voice evidence or retention policy; improving those shared inputs is usually safer and more reusable than adding a new `Voice`-only DSP branch product by product.
- For suite-wide voice improvements, the second rollout tier should be parameter-law remaps in existing `Voice` modes before any DSP-core rewrite; shared vocal context can often make dynamics, tone, proximity, subtraction, and dereverb products smarter with small policy shifts that are easy to regression-test.
- For real corpus harnesses, upmix mono spoken-word files to dual mono before running the full suite; some processors behave more realistically and more consistently with the stereo shape they normally see in hosts.
- For long-running multi-product audio-check tools, write reports incrementally after each completed product so partial results survive slow or interrupted runs.
- For voice-mode tuning outside a dedicated leveller, batch speech-corpus spread is a useful guardrail but not a guarantee that a tweak is worth keeping; if the measured change is negligible, revert the tweak and keep the harness as the asset.
- For corrective speech tools like Cleanup, add speech-band correlation and gain-matched residual metrics before concluding that a worse loudness-spread number means worse behavior; spread alone can mis-score a processor that is preserving the vocal core while removing problems.
- For learned-profile voice subtraction, broad “speech rescue” inside the subtract stage is too blunt; preserve only bins that are mid-band, voiced, and non-transient, then tune the voice-mode subtract drive around that narrower protection path.
- For voice-mode denoisers, a clean-target batch report can reveal the opposite failure mode from regressions: if the output stays extremely close to the noisy input while speech coherence is already high, increase actual clean amount and ease makeup retention before adding more protection heuristics.
- For subtle voice-mode shapers like Proximity, small policy nudges can still be worth keeping, but only when the corpus report improves on both files; if the movement is only a rounding-level average or only helps one file, treat the tier as near-saturated and move on.
- When shared vocal-context assists bias a “neutral” secondary control like body, gate that assist by the product’s actual primary processing amount or zero-amount transparency regressions will show up immediately.
- For stereo performance levellers, “linked stereo” is not enough by itself; use stereo-aware analysis or M/S-aware shaping so center-focused content can be stabilised without unnecessarily flattening the side field.
- Shared output safety must not alter listen-delta semantics: apply the emergency trimmer to normal processed output before telemetry publish, but do not trim the post-rendered listen delta or wet+listen recombination tests for subtractive products will break.
- For subtractive cleanup/corrective DSP, do not assume “only cutting” means “cannot clip”: split-band subtraction and filtered component removal can still create transient overshoot, so subtractive products need both local peak discipline and a suite-wide final output safety net.
- For analyser summary spectra, treat the first few low-frequency bands as a special case: when a log band only covers one or two FFT bins, widen the reduction neighborhood and avoid peak-dominant weighting, otherwise single-bin energy or floor jitter can produce fake low-end shape and nonsense `20 Hz` delta callouts.
- In analyzer UIs, `Avg Time` and spectral `Smoothing` are different dimensions: `Avg Time` must be continuous temporal averaging of spectrum frames, not a throttled or stepped repaint cadence, while `Smoothing` should stay a separate frequency-domain blur.
- For stage-aware analyzers, the sidebar contract must be driven by the live chain registry, not by per-stage telemetry availability; telemetry is enrichment for visible rows, not the authority on whether a VX plugin exists in the chain.
- For analyzer displays, get the tilt/slope sign exactly right: a wrong-sign display slope can make the low end look absent and the high end overemphasized even when the FFT data itself is fine.
- If a sidebar is meant to represent the active chain, do not render it from instantaneous snapshot reads alone; add a short hold/cache so brief publish jitter does not make plugins appear and disappear in the UI.
- If a sidebar depends on shared snapshot registration, give the snapshot publisher the same retry resilience as the stage publisher; a one-time registration miss should not make a plugin invisible for the entire host session.
- For analyzer displays built on JUCE real FFTs, prefer `performFrequencyOnlyForwardTransform()` when you only need magnitudes; hand-reading the packed real-transform layout is easy to get subtly wrong and can distort the visible spectrum contour.
- If a VX Suite UI control is exposed as a real processor toggle, verify the shared DSP actually consumes the corresponding param; it is easy to wire the editor and parameter layer correctly while leaving the DSP path as a silent no-op.
- In analyser-style telemetry code, prioritize hot-path wins over cosmetic cleanup: remove full-buffer copies from FFT/update loops first, then trim per-refresh allocations/string work in the editor, because those costs recur continuously while the UI is open.
- For dense analyser UIs, do not rely on one fixed horizontal control strip and fixed summary-card heights; add responsive wrapping and tolerant label scaling so the UI still reads cleanly at the product's minimum supported size.
- When a mode-specific processor feels like it does nothing, inspect the pre-DSP amount law first; an overly conservative mode multiplier can nullify an otherwise-correct DSP path before any audible processing happens.
- When an analyzer spectrum gets denser, do not keep old fixed-neighbor smoothing radii; map smoothing options by octave width or bins-per-octave, otherwise controls like `1/6 OCT` become visually inert.
- Analyzer stage rails should tolerate realistic telemetry jitter; if a freshness filter is too aggressive, the UI can collapse to misleading `Full Chain` fallback even while the chain is still live.
- If an analyzer depends on domain-scoped live telemetry, provide a visible fallback path or diagnostics when the domain scan comes up empty; otherwise the UI can look broken even though usable recent stage data still exists in the shared registry.
- For host-integrated cross-plugin telemetry, do not assume all plugins live in the same process; per-process domain lookup needs a global fallback or bridged hosts will make the analyzer appear empty.
- If an analyzer's `Avg Time` is expected to behave like SPAN, do not approximate it with a single-frame EMA alone; use a real rolling time window over recent spectrum frames so fluctuation visibly reduces while motion stays continuous.
- After shared-framework telemetry fixes, rebuild/restage the whole VX plugin chain, not just the analyser; otherwise the host can keep running old effect binaries and the analyzer symptoms will look like the fix failed.
- Once an analyser product is expected to feel like a real spectrum meter, a coarse summary spectrum is no longer sufficient; upgrade the shared telemetry FFT/band density instead of trying to fake width and range through UI smoothing, offsets, or styling.
- For correction/removal products like Deverb, do not accept large output-level loss as “just the cost of subtraction”; preserve direct-signal loudness with bounded compensation after the main removal path, and lock it in with an RMS-retention regression test.
- When users calibrate an analyser against a reference meter like SPAN, expose the key presentation controls they actually use (at minimum average time and spectrum smoothing) instead of hardcoding one editorial view.
- In analyzer UIs, never smooth sparse or narrowband spectra into the same connected broad-shape view used for voice-wide EQ changes; detect peak-dominated/test-tone cases first and switch to a discrete representation plus explicit “broad summary suppressed” language.
- In VX Suite analyser/editor UIs, do not place invisible `TextButton` hit targets over custom-painted stage rows; the shared look-and-feel will still paint the child control and can hide the row text entirely. Paint the row directly and handle hit-testing explicitly when the row itself owns the visual contract.
- For analyzer-style plots, avoid fast frame-by-frame telemetry presentation even when the backend is technically correct; slow the publish/render cadence, suppress tiny deltas, and smooth neighboring bins so the display reads as a stable explanation tool rather than jittery debug output.
- When the user asks for extraction instead of copying, prefer factoring shared logic into active VX Suite code and reusing it from the old path so legacy files visibly shrink in responsibility.
- Never leave a new VX Suite product linked to sibling-repo legacy code; import the needed implementation into this repo and improve it locally before wiring the product.
- For profile-based cleanup tools, match the product mental model literally: `Learn` should be an explicit non-destructive capture/commit flow, not a hidden auto-stop heuristic layered on top of a profile-removal UI.
- When a protection control causes audible combing or robotic tone, do not solve it with heavier dry reblending by default; prefer backing off the processor itself, especially on older ML models like `DFN2`.
- When splitting a broad voice tool into separate products, draw the boundary by job type: corrective removal in one plugin, recovery/dynamics/level in the other, and make every gain-adding path explicitly noise-aware.
- After splitting a former broad product into shared DSP plus multiple plugins, do not reason about it as a standalone VST anymore; review the shared layer by checking which wrappers still call each path and remove only the truly unreferenced branches.
- De-breath is not a generic HF smoother; gate it away from strong sibilant/transient moments so it follows real breathy exhales rather than bright consonants.
- When multiple active products own the same low-level primitive like `juce::dsp::FFT`, centralize that ownership in `Source/vxsuite/framework/` before adding more product-local analysis code on top.
- When centralizing latency-aware listen in the framework, update both the reported host latency and the internal aligned-dry buffer capacity together; otherwise the host PDC can be correct while listen auditioning is still misaligned.
- Listen semantics are product-role dependent: removal tools should audition what was removed, while additive/finishing tools should audition the added delta instead of inheriting the default removed-signal path.
- After large framework or DSP refactors, do a product-behavior pass for edge cases like Learn toggles, full-wet extreme settings, and output headroom; compile-success plus generic regression coverage is not enough.
- In causal STFT processors, do not add the reported latency twice: if the analysis FIFO/windowing already creates the intended delay, pre-offsetting the overlap-add write cursor by the same amount will silently double the real latency and break identity/PDC behavior.
- If a control is labeled `Gain`, do not let it secretly bias compressor/recovery policy upstream. Keep it semantically honest as a neutral-centered final gain stage unless the product explicitly says otherwise.
- When shared editor hit targets are drawn in editor coordinates, convert mouse events with `getEventRelativeTo(this)` before testing bounds; child-local positions silently break inline controls.
- Do not use output trimming as the primary fix for additive/EQ-style VX products; first reduce the upstream gain law or stabilize the DSP itself, then keep any trimmer strictly as an emergency guard.
- In latency-bearing stereo processors, never add a post-stage dry recovery blend unless it is proven sample-aligned with the wet path and product intent; otherwise it reads as stereo doubling or delay even when the core DSP is correct.
- In finishing/master-style dynamics, avoid soft clipping as the default limiter behavior. Leave explicit pre-limiter headroom and use clean gain limiting first; otherwise the limiter becomes the distortion source instead of the protection stage.
- When a compressor param is set to 0.0, the gain computation must also be bypassed (guard with `compAmt > ε`); threshold/ratio at compAmt=0 are not identity and will cause unintended gain reduction.
- In M/S spectral processors, dynamic side-signal scaling (sideScale ≠ 1.0) to maintain M/S energy balance causes time-varying stereo width modulation that is perceived as stereo delay/phasing; leave sideScale = 1.0 and accept the M/S balance as-is.
- In the framework `ProcessCoordinator`, always capture dry into the listen buffer regardless of whether listen mode is on; the latency-aligned dry buffer is also needed by processors (e.g. Deverb body restore) that are not listen-specific, and skipping capture when listen is off silently breaks those non-listen paths.
- When a causal STFT processor pre-seeds `olaWritePos = latencySamples` in `reset()`, the FIFO already provides that same latency, so the write offset doubles the true end-to-end delay while reported latency stays at `kFftSize - kHop`; always start `olaWritePos = 0` and let the FIFO provide the latency.
- In REAPER Lua, do not assume API functions follow the same multi-return pattern as Python docs or older snippets; validate return arity before concatenating values, especially for calls like `TrackFX_GetUserPresetFilename`.
- For VX activity lights, make zero-amount defaults explicitly idle and drive the LED from real engaged processing, not from a still-live fixed-threshold stage that happens to sit behind a "0" knob position.
- When mapping a bipolar normalized control around center, do not use the 3-argument `juce::jmap`; it treats the input as 0..1 and will silently shift a neutral `0.5` control off-center.
- When the user asks to replace an analyzer product with a new stage-aware design, do not carry the old analyzer UI forward under a new name; replace the interaction and render model itself so the product matches the agreed explanation contract.
- Analyzer visuals are only as good as their semantic contract: before making plots prettier, lock the cadence, units, smoothing, axes, and dry-baseline behavior so the graph reads like a familiar EQ/dynamics tool instead of debug telemetry.
- For analyzer stage rails, do not paint informative row content underneath live child controls; either let the row component own its text or handle hit-testing separately, otherwise plugin names can disappear even when the data model is correct.
- For VX Suite mode behavior, treat `Vocal` as a shared protection contract across products: preserve common vocal ranges, avoid EQ/dynamics moves that damage intelligibility, and avoid reintroducing noise; product-specific exceptions like DeepFilterNet model selection or Deverb-triggered WPE are acceptable, but the rest of the suite should derive from one consistent definition rather than bespoke per-plugin interpretations.
- If a denoiser starts reading like broadband level loss, check whether the noise estimator is chasing program material and whether the speech-protection fields from the framework are actually used in the DSP; mode-specific policy is not real unless it reaches the gain law itself.
- For denoisers with a vocal-protection mode, never apply speech-band protection or loudness-retention blindly to noise-only input; gate those protections by actual speech presence, or pink noise and broadband noise tests will look like volume-drop-or-near-bypass failures instead of real denoising.
- For analyser spectrum views, avoid switching between fundamentally different render metaphors mid-stream; a brief fallback into sparse stems/bars is more confusing than helpful, so keep the main overlay render stable and use sparse detection only to influence text/annotation unless the user explicitly wants a discrete-band view.
- For framework-based plugin UIs, solve readability and resize behavior in the shared editor layer first: titles, status text, knob labels, and hint copy should inherit tolerant minimum scales and roomier layout rules from `EditorBase` rather than being fixed later product by product.
- When a Cleanup-style corrective processor computes voiced-material protection signals upstream, make sure the DSP gain law actually uses them; otherwise the UI/framework can claim “speech-aware cleanup” while the live corrective stage still over-chews bright voiced material and reads as distortion.
- In serial subtractive stages, never derive a later band-split subtraction from a stale pre-processed lowpass state after an earlier stage has already changed the sample; recompute the split from the current sample state or the recombination becomes distortion-prone.
- If a user reports a specific activity light firing at the moment of distortion, add a regression that measures that activity path directly; output-only regressions can miss the exact false detector that is causing the audible bug.
- For plosive detection in voice cleanup, do not let the detector track the same low-mid/voiced region used for normal pitch energy, and do not keep plosive strength high at presence/air-biased settings; otherwise voiced syllables can be misread as plosives and the meter will truthfully expose the bug.
- For Cleanup-style voiced-distortion regressions, do not rely on peak/headroom checks or one forgiving harmonic-tone case; add a brighter voiced edge-case and assert coherence plus best-gain residual so “still sounds crunchy” reports are caught even when the signal stays finite and under ceiling.
- In JUCE/VST processor reporting, never treat latency as tail by default. PDC delay and post-input carryover are separate host contracts, and tests should only expect rendered tail audio from processors that explicitly report non-zero tail length for a real decay/carryover path.
- When a distortion report narrows to a specific optional tone-shaping switch like Cleanup’s high shelf, compare the exact same signal with the switch off versus on; that isolates whether the bug is in the optional stage itself or just in the broader processor setting.
- For adaptive filter stages like Cleanup’s high shelf, do not zero the biquad state every time the coefficients retune under live program-dependent control, and keep the gain law modest on voiced material; aggressive per-block retunes plus state resets are a reliable recipe for “crunchy when enabled” reports.
- For stereo noise tools, do not collapse the program to one shared mono/mid denoise path and then reconstruct stereo around it when left/right can contain different noise. Learned profiles, readiness gates, and fallback paths all need to be channel-aware, and an untouched channel should use a latency-aligned dry path rather than traveling through the denoise FFT just because its sibling is active.
- In the shared VX editor/framework, analyzer-style visuals like level traces should be opt-in per product, not suite-wide by default; if a visual only meaningfully supports one product’s promise, leaving it everywhere creates confusion and makes timing mismatches much more obvious.
- For `Leveler`, a full “smart rider” rewrite that looks architecturally cleaner is not automatically better; keep the proven broad gain-rider core if it still wins on the real file and regressions, then push aggression through a narrow top-end intervention zone instead of replacing the whole control law in one leap.
- For `Leveler`, loudness-led research was still the right architectural direction, but a first implementation can easily outperform on one real file while failing short synthetic hot-mix regressions; do not ship the new rider core until it wins on both the real workflow file and the regression lane.
- For `Leveler`, the safer migration path is mode-by-mode: keep the proven vocal engine intact, introduce the loudness-led architecture first in `Mix Leveler`, and only then rework `Vocal Rider` once the split architecture itself is verified.
- For `Vocal Rider`, phrase-aware speech context is most useful as a local anchor and intervention bias, not as a direct multiplier on the main ride target; pushing phrase/intelligibility bias into the core gain target can look smarter but regress the real file, while support-lane bias keeps the improvement and the regression safety.
- For `Mix Leveler`, the useful perceptual split is `L_m / L_s / L_b`: let short-term minus baseline drive the slow rider, let momentary minus short-term drive the spike clamp, and keep the deadband explicit around about `1 LU`; trying to “recenter” a weaker envelope rider afterward is the wrong fix and will usually fail the hot-mix regressions.
- When a VX dynamics product still looks or feels too quiet in host use after a real DSP improvement, check for duplicated safety stages before retuning the core law; keep the simplification if found, but verify with file measures because removing redundant trimming may have zero audible/metric effect if the trimmer was not engaging.
- For `Leveler`, spread reduction alone is not a sufficient tuning target; add RMS and peak reporting to the real-file harness and judge `Mix Leveler` on a three-way tradeoff between spread, absolute loudness retention, and regression safety, because the strongest spread result can still be the wrong host-feel build.
- For `Mix Leveler`, the safe way to recover believable output level is not a generic recentering or makeup stage; add a separate normalization layer that compares dry vs wet baseline loudness, cap it tightly, and gate it by both headroom and active spike control so it supports the rider instead of undoing it.
- Once `Leveler` reaches the “tradeoff wall,” stop hand-tweaking and automate the search: expose a small bounded tuning surface for `Mix`, score candidates against the real dry file plus the best host-approved reference render, and only promote a candidate that still passes the regression suite.
- For products that need a second user-facing choice beyond `Mode`, extend the shared framework with a generic auxiliary selector instead of building one-off editor wiring; that keeps the UI consistent and lets new mode families like `Mix Leveler` analysis modes ship without forking the framework.
- For `Leveler` V3, true offline analysis inside the plugin is constrained by host source access; implement the real offline target path and a clean fallback to Smart Realtime first, then use offline-capable harnesses to validate the analysis path instead of faking pre-scan behavior in the realtime plugin.
- For `Leveler` Smart startup tuning, do not guess from the waveform alone; add a lightweight harness trace for the internal `Mix` state first, because the real startup bug may be baseline poisoning from silent intro material rather than a confidence or makeup problem.
- For `Mix Leveler`, do not let near-silent opening material prime the loudness engine. Keep the delayed dry path until there is real activity, or the baseline will start far too low and the rider/spike clamp will over-attenuate the first audible section.
- When a measurement harness offers both “trace” and “plain render” paths, never reuse the same processor instance for both. Trace/debug passes can leave state behind and make a rendered reference look better or worse than the actual plugin behavior.
- For `Leveler`, if a more “product-grade” control-law refinement makes Smart mode read like a broad level drop again on the real file, reject it even if the architecture sounds cleaner on paper. Keep the proven priming fix and measurement improvements instead of shipping a smarter-looking but less musical rider.
- For `Leveler`, error blending and dynamic-range-aware control are not automatically wins just because they are conceptually cleaner. On real mixed material, introduce them one seam at a time and compare against the current host-approved baseline, because the first drop-in version can easily over-tighten the intro and read as blanket attenuation.
- When a user frames `Leveler` as an automated human fader, keep any whole-track restore stage fader-like and mode-specific: use long-horizon programme restoration only to stop `Mix Leveler` ending up globally quieter than the source, and do not let that restore path leak into `Vocal Rider` if it weakens real consistency gains.
- For capture/analyse UX, never show a single `%` meter that silently switches meaning between “enough coverage to stop” and “final confidence.” While capture is running, label it as `Coverage`; only show `Confidence` after the map is locked, or users will reasonably assume `100%` means analysis is complete.
- For offline-analysis confidence, do not score a valid full-track capture like a fragile quality heuristic. Start from a “usable map” floor once coverage is good, then add smaller duration/range bonuses; otherwise a normal successful analysis can come back at ~50% and undermine trust in the feature.
- For heuristic source-rebalance DSP, do not let broad residual lanes like `Guitar` or low-end continuity lanes like `Bass` become the default owners of the spectrum. Stronger semantic lanes such as `Vocals` and `Drums` must claim their expected regions first, or most sliders will appear to move the same material and the product will feel fundamentally mis-mapped.
- For mastered stereo source rebalance, accept the identifiability limit early: once repeated heuristic retunes still make `Vocals` or other lanes swallow too much of the spectrum, static STFT priors are no longer enough and the product should pivot to ML-guided source ownership rather than pretending frequency- and transient-only masks can separate overlapping sources cleanly.
- In realtime spectral DSP, never drain an output FIFO by shifting the whole buffer one sample at a time. Even if the first prototype “works,” that O(n)-per-sample pattern will burn CPU immediately; use a circular buffer or `AbstractFifo` from the start.
- If a DSP object receives control targets or analysis context through setter calls that may conceptually come from outside the audio loop, publish them through an explicit lock-free handoff and smooth them inside the DSP path. Even when the current caller happens to be the audio thread, making the ownership explicit prevents fragile hidden assumptions.
- When a detector concept proves broadly useful across products, promote only the reusable evidence layer into the framework and keep the behavioural response local to each product. Shared `SignalQuality`-style analysis belongs beside `VoiceAnalysis` / `VoiceContext`; product-specific weighting laws do not.
- For semantic rebalance tools, neutral settings must short-circuit to exact dry pass-through. Do not rely on a heuristic STFT round-trip to “probably” be transparent at 0 dB; if the product contract says neutral, implement an explicit neutral bypass path first and only then tune the active processing range.
- For multi-lane DSP products, never rely on implicit array ordering between UI controls and internal source masks. Define explicit source indices or enums and use them consistently end-to-end, or you can spend hours “tuning” what is really just a lane-mapping bug.
- When a heuristic DSP product needs to behave differently on `Studio`, `Live`, and `Phone / Rough` material, do not bury those differences as scattered conditionals inside one mask function. Promote them into explicit data-driven mode profiles with user-facing selection, shared parameter plumbing, and clear per-mode safety limits.
- When a source-profile table includes split ownership regions like `low + high drums` or separate medium vocal zones, do not compress it into a single contiguous band model. The profile data structure must be able to encode the table directly, or the implementation will silently flatten into the wrong spectral ownership.
- When a product’s control range changes, immediately update any measurement harnesses and offline profilers that map dB to normalized values. Stale harness ranges make later tuning and review results look precise while testing the wrong contract.
- For ML-product specs, do not stop at the clean architecture boundary. Name the actual delivery risks explicitly: dataset quality, runtime/backend choice, and latency mode. A spec that gets the model/DSP split right but leaves those three points vague is still not implementation-ready.
- When wiring a native ML runtime to an exported ONNX model, validate the real tensor contract from the exported artifact before writing the C++ runner. A silent frame-count mismatch between export and runtime is enough to corrupt memory and masquerade as a generic runtime crash.
- If a lightweight imported ONNX Runtime dylib plus vendored headers behaves unpredictably with the C++ wrapper API, prefer the stable C API boundary. It is a better fit for plugin/runtime integration and avoids C++ wrapper/ABI surprises in mixed local environments.
- When the repo already contains a proven ML/runtime pattern, reuse that shape first. DeepFilterNet’s model discovery, imported-runtime setup, resource handling, and resampling strategy should be the baseline for new ML products before inventing a parallel integration path.
