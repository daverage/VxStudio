# Lessons

- Shared output safety must not alter listen-delta semantics: apply the emergency trimmer to normal processed output before telemetry publish, but do not trim the post-rendered listen delta or wet+listen recombination tests for subtractive products will break.
- For subtractive cleanup/corrective DSP, do not assume “only cutting” means “cannot clip”: split-band subtraction and filtered component removal can still create transient overshoot, so subtractive products need both local peak discipline and a suite-wide final output safety net.
- For analyser summary spectra, treat the first few low-frequency bands as a special case: when a log band only covers one or two FFT bins, widen the reduction neighborhood and avoid peak-dominant weighting, otherwise single-bin energy or floor jitter can produce fake low-end shape and nonsense `20 Hz` delta callouts.
- In analyzer UIs, `Avg Time` and spectral `Smoothing` are different dimensions: `Avg Time` must be continuous temporal averaging of spectrum frames, not a throttled or stepped repaint cadence, while `Smoothing` should stay a separate frequency-domain blur.
- For stage-aware analyzers, the sidebar contract must be driven by the live chain registry, not by per-stage telemetry availability; telemetry is enrichment for visible rows, not the authority on whether a VX plugin exists in the chain.
- For analyzer displays, get the tilt/slope sign exactly right: a wrong-sign display slope can make the low end look absent and the high end overemphasized even when the FFT data itself is fine.
- If a sidebar is meant to represent the active chain, do not render it from instantaneous snapshot reads alone; add a short hold/cache so brief publish jitter does not make plugins appear and disappear in the UI.
- If a sidebar depends on shared snapshot registration, give the snapshot publisher the same retry resilience as the stage publisher; a one-time registration miss should not make a plugin invisible for the entire host session.
- For analyzer displays built on JUCE real FFTs, prefer `performFrequencyOnlyForwardTransform()` when you only need magnitudes; hand-reading the packed real-transform layout is easy to get subtly wrong and can distort the visible spectrum contour.
- If a VX Suite UI control is exposed as a real processor toggle, verify the shared DSP actually consumes the corresponding param; it is easy to wire the editor and parameter layer correctly while leaving the DSP path as a silent no-op.
- In analyser-style telemetry code, prioritize hot-path wins over cosmetic cleanup: remove full-buffer copies from FFT/update loops first, then trim per-refresh allocations/string work in the editor, because those costs recur continuously while the UI is open.
- For dense analyser UIs, do not rely on one fixed horizontal control strip and fixed summary-card heights; add responsive wrapping and tolerant label scaling so the UI still reads cleanly at the product's minimum supported size.
- When a mode-specific processor feels like it does nothing, inspect the pre-DSP amount law first; an overly conservative mode multiplier can nullify an otherwise-correct DSP path before any audible processing happens.
- When an analyzer spectrum gets denser, do not keep old fixed-neighbor smoothing radii; map smoothing options by octave width or bins-per-octave, otherwise controls like `1/6 OCT` become visually inert.
- Analyzer stage rails should tolerate realistic telemetry jitter; if a freshness filter is too aggressive, the UI can collapse to misleading `Full Chain` fallback even while the chain is still live.
- If an analyzer depends on domain-scoped live telemetry, provide a visible fallback path or diagnostics when the domain scan comes up empty; otherwise the UI can look broken even though usable recent stage data still exists in the shared registry.
- For host-integrated cross-plugin telemetry, do not assume all plugins live in the same process; per-process domain lookup needs a global fallback or bridged hosts will make the analyzer appear empty.
- If an analyzer's `Avg Time` is expected to behave like SPAN, do not approximate it with a single-frame EMA alone; use a real rolling time window over recent spectrum frames so fluctuation visibly reduces while motion stays continuous.
- After shared-framework telemetry fixes, rebuild/restage the whole VX plugin chain, not just the analyser; otherwise the host can keep running old effect binaries and the analyzer symptoms will look like the fix failed.
- Once an analyser product is expected to feel like a real spectrum meter, a coarse summary spectrum is no longer sufficient; upgrade the shared telemetry FFT/band density instead of trying to fake width and range through UI smoothing, offsets, or styling.
- For correction/removal products like Deverb, do not accept large output-level loss as “just the cost of subtraction”; preserve direct-signal loudness with bounded compensation after the main removal path, and lock it in with an RMS-retention regression test.
- When users calibrate an analyser against a reference meter like SPAN, expose the key presentation controls they actually use (at minimum average time and spectrum smoothing) instead of hardcoding one editorial view.
- In analyzer UIs, never smooth sparse or narrowband spectra into the same connected broad-shape view used for voice-wide EQ changes; detect peak-dominated/test-tone cases first and switch to a discrete representation plus explicit “broad summary suppressed” language.
- In VX Suite analyser/editor UIs, do not place invisible `TextButton` hit targets over custom-painted stage rows; the shared look-and-feel will still paint the child control and can hide the row text entirely. Paint the row directly and handle hit-testing explicitly when the row itself owns the visual contract.
- For analyzer-style plots, avoid fast frame-by-frame telemetry presentation even when the backend is technically correct; slow the publish/render cadence, suppress tiny deltas, and smooth neighboring bins so the display reads as a stable explanation tool rather than jittery debug output.
- When the user asks for extraction instead of copying, prefer factoring shared logic into active VX Suite code and reusing it from the old path so legacy files visibly shrink in responsibility.
- Never leave a new VX Suite product linked to sibling-repo legacy code; import the needed implementation into this repo and improve it locally before wiring the product.
- For profile-based cleanup tools, match the product mental model literally: `Learn` should be an explicit non-destructive capture/commit flow, not a hidden auto-stop heuristic layered on top of a profile-removal UI.
- When a protection control causes audible combing or robotic tone, do not solve it with heavier dry reblending by default; prefer backing off the processor itself, especially on older ML models like `DFN2`.
- When splitting a broad voice tool into separate products, draw the boundary by job type: corrective removal in one plugin, recovery/dynamics/level in the other, and make every gain-adding path explicitly noise-aware.
- After splitting a former broad product into shared DSP plus multiple plugins, do not reason about it as a standalone VST anymore; review the shared layer by checking which wrappers still call each path and remove only the truly unreferenced branches.
- De-breath is not a generic HF smoother; gate it away from strong sibilant/transient moments so it follows real breathy exhales rather than bright consonants.
- When multiple active products own the same low-level primitive like `juce::dsp::FFT`, centralize that ownership in `Source/vxsuite/framework/` before adding more product-local analysis code on top.
- When centralizing latency-aware listen in the framework, update both the reported host latency and the internal aligned-dry buffer capacity together; otherwise the host PDC can be correct while listen auditioning is still misaligned.
- Listen semantics are product-role dependent: removal tools should audition what was removed, while additive/finishing tools should audition the added delta instead of inheriting the default removed-signal path.
- After large framework or DSP refactors, do a product-behavior pass for edge cases like Learn toggles, full-wet extreme settings, and output headroom; compile-success plus generic regression coverage is not enough.
- In causal STFT processors, do not add the reported latency twice: if the analysis FIFO/windowing already creates the intended delay, pre-offsetting the overlap-add write cursor by the same amount will silently double the real latency and break identity/PDC behavior.
- If a control is labeled `Gain`, do not let it secretly bias compressor/recovery policy upstream. Keep it semantically honest as a neutral-centered final gain stage unless the product explicitly says otherwise.
- When shared editor hit targets are drawn in editor coordinates, convert mouse events with `getEventRelativeTo(this)` before testing bounds; child-local positions silently break inline controls.
- Do not use output trimming as the primary fix for additive/EQ-style VX products; first reduce the upstream gain law or stabilize the DSP itself, then keep any trimmer strictly as an emergency guard.
- In latency-bearing stereo processors, never add a post-stage dry recovery blend unless it is proven sample-aligned with the wet path and product intent; otherwise it reads as stereo doubling or delay even when the core DSP is correct.
- In finishing/master-style dynamics, avoid soft clipping as the default limiter behavior. Leave explicit pre-limiter headroom and use clean gain limiting first; otherwise the limiter becomes the distortion source instead of the protection stage.
- When a compressor param is set to 0.0, the gain computation must also be bypassed (guard with `compAmt > ε`); threshold/ratio at compAmt=0 are not identity and will cause unintended gain reduction.
- In M/S spectral processors, dynamic side-signal scaling (sideScale ≠ 1.0) to maintain M/S energy balance causes time-varying stereo width modulation that is perceived as stereo delay/phasing; leave sideScale = 1.0 and accept the M/S balance as-is.
- In the framework `ProcessCoordinator`, always capture dry into the listen buffer regardless of whether listen mode is on; the latency-aligned dry buffer is also needed by processors (e.g. Deverb body restore) that are not listen-specific, and skipping capture when listen is off silently breaks those non-listen paths.
- When a causal STFT processor pre-seeds `olaWritePos = latencySamples` in `reset()`, the FIFO already provides that same latency, so the write offset doubles the true end-to-end delay while reported latency stays at `kFftSize - kHop`; always start `olaWritePos = 0` and let the FIFO provide the latency.
- In REAPER Lua, do not assume API functions follow the same multi-return pattern as Python docs or older snippets; validate return arity before concatenating values, especially for calls like `TrackFX_GetUserPresetFilename`.
- For VX activity lights, make zero-amount defaults explicitly idle and drive the LED from real engaged processing, not from a still-live fixed-threshold stage that happens to sit behind a "0" knob position.
- When mapping a bipolar normalized control around center, do not use the 3-argument `juce::jmap`; it treats the input as 0..1 and will silently shift a neutral `0.5` control off-center.
- When the user asks to replace an analyzer product with a new stage-aware design, do not carry the old analyzer UI forward under a new name; replace the interaction and render model itself so the product matches the agreed explanation contract.
- Analyzer visuals are only as good as their semantic contract: before making plots prettier, lock the cadence, units, smoothing, axes, and dry-baseline behavior so the graph reads like a familiar EQ/dynamics tool instead of debug telemetry.
- For analyzer stage rails, do not paint informative row content underneath live child controls; either let the row component own its text or handle hit-testing separately, otherwise plugin names can disappear even when the data model is correct.
- For VX Suite mode behavior, treat `Vocal` as a shared protection contract across products: preserve common vocal ranges, avoid EQ/dynamics moves that damage intelligibility, and avoid reintroducing noise; product-specific exceptions like DeepFilterNet model selection or Deverb-triggered WPE are acceptable, but the rest of the suite should derive from one consistent definition rather than bespoke per-plugin interpretations.
- If a denoiser starts reading like broadband level loss, check whether the noise estimator is chasing program material and whether the speech-protection fields from the framework are actually used in the DSP; mode-specific policy is not real unless it reaches the gain law itself.
- For denoisers with a vocal-protection mode, never apply speech-band protection or loudness-retention blindly to noise-only input; gate those protections by actual speech presence, or pink noise and broadband noise tests will look like volume-drop-or-near-bypass failures instead of real denoising.
- For analyser spectrum views, avoid switching between fundamentally different render metaphors mid-stream; a brief fallback into sparse stems/bars is more confusing than helpful, so keep the main overlay render stable and use sparse detection only to influence text/annotation unless the user explicitly wants a discrete-band view.
- For framework-based plugin UIs, solve readability and resize behavior in the shared editor layer first: titles, status text, knob labels, and hint copy should inherit tolerant minimum scales and roomier layout rules from `EditorBase` rather than being fixed later product by product.
